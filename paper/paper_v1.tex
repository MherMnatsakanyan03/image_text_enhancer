\subsection{Background}
\label{sec:background}

The extraction of textual information from images of scanned books to even receipts, has become an important part of everyday life \cite{alqudah_2015,review_degradation_image_enhancement_zhou_2023}. Any such extraction necessarily begins with the pre-processing of recorded images through techniques such as binarisation, noise removal, and geometric transformation, a process generally referred to as \textit{document enhancement}. Numerous tools focused on this niche have been proposed. Although these approaches feature broad capabilities, they are often associated with substantial complexity on the user's end and entail considerable software dependencies. These dependencies restrict applicability, particularly in settings where binary size is constrained, such as mobile and web platforms. This limitation assumes added importance in a context where most digital media interactions now take place on smartphones and application development is increasingly oriented toward edge-computing models. Consequently, efficient and platform-independent implementations that exploit parallelisation and other optimisation techniques are of considerable importance for reducing processing time and enhancing user satisfaction.

Image enhancement and document preprocessing are covered by a wide range of open-source tools and commercial software. The scope ranges from general purpose image processing and OCR libraries to specialised tools for post-processing scanned documents.

ImageMagick \cite{imagemagick} is a suite for image processing that is available as a command line tool and as a library. It offers functions such as format conversion, resizing, filtering, and colour correction. However, ImageMagick is not specifically designed to enhance scanned or photographed images. Similar to OpenCV, the diverse features result in a large code base.

The C-based Leptonica library \cite{leptonica} is a general-purpose image processing and image analysis application. It is a dependency to both Tesseract and OpenCV. Although Leptonica is relatively lightweight compared to the other libraries, much like our approach, it primarily serves as a low-level library and does not offer ready-to-use, configurable processing pipelines.

\label{sec:specialised-post-processing}
A very similar tool to our approach is Unpaper \cite{unpaper}, which focuses on post-processing scanned documents. It offers features such as deskewing, binarisation, and noise reduction that have been developed specifically for this purpose. The tool also offers a CLI variant, with which pipeline steps can be configured and disabled. Unpaper requires the library ffmpeg, which in the standard configuration consumes >100MiB of space.

\subsection{Our Contribution}
\label{sec:our-contribution}

Unlike the existing tools and libraries described, our approach is designed to be lightweight, extensible, and user-friendly. Instead of being a comprehensive image processing library or OCR engine, we focus on a lightweight, modular image enhancement pipeline, focused on printed and handwritten text, written entirely in C++ with only \href{https://www.cimg.eu/}{CImg} as a dependency, which is itself is lightweight and has no further dependencies \cite{cimg}. Image Text Enhancer (ITE) has a lightweight build footprint, requiring only three build dependencies: OpenMP, X11, and JPEG. It can be imported as a library and also as a CLI executable---with 1.7 MiB on Linux.

Compared to OpenCV and ImageMagick, our work is characterised by its deliberate focus on document enhancement and compactness. Unlike Leptonica, our pipeline is specialised in the post-processing of scanned documents and unlike Unpaper, our binary including dependencies is $\approx$50$\times$ smaller.
In addition, our licensing model (MIT-0 and CeCILL-C) permits use in research and commercial applications. By combining minimal dependencies, a modular architecture, portability through C++ and OpenMP, and a commercially usable licence, our pipeline fills a gap in the current landscape.

We present our library, describe optimsiation choices for our pipeline and address the question and address challenges in the optimisation of a document enhancement pipeline. We then evaluate the performance gains through various experiments. In a separate proof of concept, we show that our approach can be compiled for WASM and integrated into a \underline{\href{https://ite.daniel-motz.de}{web application}}. % todo: decide whether to remove the WASM PoC

\section{Pipeline}
\label{sec:pipeline}
  
The pipeline is designed to be modular, enabling users to activate, deactivate and configure each stage as required, both through the CLI and the library. Some stages can intentionally not be disabled. These exceptions are marked in algorithm \ref{alg:enhance} with an \textbf{asterisk}. 

\begin{algorithm}
  \caption{Image Text Enhancer Pipeline}
  \label{alg:enhance}
  \begin{enumerate}
  \item Grayscale Conversion\textbf{*}
  \item Deskew
  \item Contrast Enhancement
  \item Denoise: (Adaptive) Gaussian and Median Blur
  \item Binarisation\textbf{*}: Otsu, Sauvola, Bataineh
  \item Morphological Operations
  \item Colour passthrough
\end{enumerate}
\end{algorithm}

The user can choose between different algorithms for certain stages---as these have different effects on the result, users can select the algorithm best suited to their application. For instance, Bataineh's adaptive binarisation method with automatic window adjustment \cite{adaptive_thresholding_methods_bataineh_2011} is helpful when binarising historical, handwritten documents with uneven writing and lighting, whereas Otsu's method \cite{Otsu1979} is a good choice for binarising typewritten documents with uniform font and lighting.
The pipeline options and standard configuration of the pipeline are documented in the help print of the CLI application---the user only adds the input and output file.
\begin{verbatim}
      ./ite -i music-sheet.jpg -o enhanced.jpg
\end{verbatim}
The standard configuration delivers good results for most use cases and is based on recommendations by the authors. This means that the pipeline can be applied immediately. However, users can also customise the configurations to optimise the pipeline and improve their specific results.

We provide a command-line application that can be used independently of a program to access the pipeline. This enables the pipeline to be configured and used directly via the command line.

We hypothesise that the baseline implementation fails to scale linearly with problem size, although such scaling would be desirable for image processing workloads. We therefore concentrated on identifying low-hanging fruits at the level of high-level algorithmic design, for example by selecting more efficient algorithms for specific tasks, while aiming to adhere to best practices such as avoiding false sharing and writing code amenable to auto-vectorisation. For the purposes of the paper, we focus on the stages, whose optimisations had the biggest effect. The full list of applied optimisations is documented in the project's repository. % todo: add Optimizations.txt to the git repository.

\subsection{Deskew}
Instead of using a "rotate-and-project" approach, which repeatedly rotates the image to find an appropriate angle, we applied the Radon transform, which gives an acceptable rotation after the first application. Also, we removed an overlook, where the deskew operation was applied to the grayscale and colour image, if the colour passthrough option was enabled.  % todo: explain why Radon transform gives an acceptable solution if I want to

\subsection{Bataineh}
The main feature and bottleneck of the Bataineh algorithm is the adaptive sliding window, which takes into account the number of pixels from three categories---background, foreground and confusion \cite{adaptive_thresholding_methods_bataineh_2011}. This requires summing over areas of the image with masks reflecting these categories. The baseline approach was to sum using a nested for loop for each window. The optimised variant uses two integral images for background and confusion pixels, yielding a significant complexity reduction. We estimate our baseline implementation with $O(n \cdot w^2)$---where $n$ is the number of pixels and $w$ the window size---while the improved implementation is in $O(n)$.

\subsection{Contrast Enhancement}

The contrast enhancement stage simply linearises the histogram. The baseline implementation was not parallelised, which was the first optimisation. Also, the lookup-table is pre-computed, which allows for auto-vectorisation in the application of the lookup-table to the image. Similarly, the Sauvola binarisation method, uses a branchless lookup in the optimised variant.

\subsection{Morphology}
The pipeline only supports dilation and erosion, which can both be realised by linear filters. The baseline implementation used four nested for-loops\footnote{for colour images it is five, as it is done for each channel, but our approach is only used on binary images} for iterating the image and scanning the neighbourhood. Since the filter is separable, it is divided into a horizontal and vertical pass. Instead of scanning the neighbourhood, a queue is used to compute the min/max of a given 1D window in amortised $O(1)$.

\section{Experiments}
\label{sec:experiments}

To evaluate the performance of the pipeline with the optimisations, various experiments are conducted and the results are compared with the baseline implementation. The experiments examine the runtime when processing images of different sizes with and without parallelisation.

For our measurements, we use a machine with an AMD Ryzen 9950X3D 16-core processor (32 hyperthreads) running WSL2 with Ubuntu 24.04.1 LTS with 64 GB of RAM\footnote{It is clear that this comes with a performance penalty, however we can expect an even additive penalty across our conducted tests. The profiling benchmarks were conducted on native Ubuntu 24.04.1 LTS. Each core has a base frequency of 4.3 GHz, with boost frequency of around 5.5 GHz. Care has been taken to limit the number of background processes, so as not to affect performance.

We selected a small but plausible set of document images spanning both document type and resolution. The benchmark includes textual documents of various sizes and degrees of detail. We will refer to the images in terms of their width in pixels, i.e. 1K or 16K (for details, see \href{https://github.com/MherMnatsakanyan03/image_text_enhancer/tree/main/resources/test_images}{the repository}). %(see Table \ref{tab:optimised-runtime-8cores} for details). 
These choices aim to capture common document-enhancement scenarios while also testing scalability to very large inputs.

The pipeline was configured to apply the following operations: deskew, adaptive gaussian, simple median filter, adaptive median filter, Bataineh binarisation, despeckle, dilation, erosion and color pass. All these stages were also timed individually. The benchmarks do not report the time necessary for loading and writing the image to and from disk.

The optimisations significantly reduce runtime, especially for larger inputs. The optimised implementation scales nearly linearly, with some overhead, i.e. serial regions. Table \ref{tab:phase-runtime} points to two main improvements: \textbf{Deskew} and \textbf{Binarisation (Bataineh)} which are individually $99\%$ faster. 
However, the runtime scaling benchmark does not tell us about the use of parallel resources. We therefore use a strong-scaling benchmark on a fixed problem size. We tested 1 to 32 threads. In Figure \ref{fig:strong-scaling-main-vs-optimised} we observe a well-behaved speedup of the baseline implementation, whereas the optimised code scales worse, where a 32-fold thread increase results in roughly 4-fold speedup. This suggests that there are serial regions in the program, that dominate after the optimisations have been applied, or, that the CPU speed is bound by waiting on e.g. main memory. It is also expected, that the scaling is slightly better for images with a larger problem size, namely because unavoidable serial regions like I/O take up less of the program's execution time. % todo: the benchmarks do not include the I/O time. Find an explanation, why the benchmark behaves better on larger images. 

For more information on the bottlenecks, we created a profile using AMDuProf over an 8K image (find detailed profiling reports in the repository). % todo: add them to the repository
Figure \ref{fig:uprof_baseline} shows the profile plot of the \textbf{baseline implementation}. The plot is dominated by the Bataineh algorithm (the large plateau in the middle), meaning it makes good use of parallel resources---most time is spent on the summing of the area, which is indeed highly parallelised. This is also reflected in Table \ref{tab:phase-runtime}, where the Bataineh binarisation has $95\%$ of total runtime attributed to it. This explains the good strong scaling performance and also why the optimised variant is so much quicker.

The \textbf{optimised profile} shows a different pattern---very few truly parallel regions can be spotted. The plot shows $\approx 27\%$ I/O, which however are not further discussed, as it is excluded in the benchmark, as well as large serial regions and memory allocations inside of pipeline stages, which can be attributed to e.g. allocation of stage-local scratch buffers for the morphological functions, masks for the Bataineh algorithm and even the output image for the binarisation functions. From the profiling and associated Assembly, we estimate that $15\%$ of the total pipeline time (on an 8K image in a single serial profiling run) was spent on such memory operations.
Another caveat are the CImg library functions---both the gaussian and median filter appear serial, even when forcing CImg to use OpenMP. The single largest impact in this benchmark scenario is the median filter, with $\approx 15.1\%$ serial execution on an 8K image. The theoretical speedup $S_\text{max}$ is limited by the serial regions of the program. Our observed speedup is however $4.76$ (see Figure \ref{fig:strong-scaling-main-vs-optimised}), implying, that these regions are not fully serial.
\begin{align*}
  S_\text{max} \approx \frac{1}{0.301} \approx 3.32 <  4.76 \approx \frac{1}{0.21} \approx S_\text{observed}
\end{align*}

With this hypothesis, we introduced another optimised variant, where the caller allocates scratch and output buffers, meaning they can be reused throughout the pipeline. With this we found a speedup of $6.16$ on an 8K image and 32 threads.

\subsection{Conclusions}

The experiments show that our hypothesis, that algorithmic improvements were important for improving the runtime, is true. Compared with the baseline, the optimised implementation achieves nearly linear runtime growth with image size and reduces total runtime by up to $97.8\%$ on some images. However, the strong-scaling benchmark shows, that the main reason the optimised variant is faster, are the algorithmic optimisations and resulting lower work complexity. Applying the optimisations exposed new ceilings, which are for one memory related and also ineffective use of library functions. Also, the evaluation is incomplete, as only a fraction of e.g. binarisation algorithms were evaluated. Therefore, further work is necessary to make this library production ready.

\section{Future Work}
\label{sec:future-work}

Possible future work is grouped into three dimensions: visual (result) performance improvements, implementation improvements and benchmarking. For one, the visual performance of the library was not systematically evaluated in this paper, as this was not the focus of this lecture. Also, the benchmark was limited to only the Bataineh binarisation algorithm, leaving out the also implemented Otsu and Sauvola, which have very different behaviour. In future work the WASM runtime performance could be evaluated, while also aiming to improve the performance by using intrinsics or rewriting the code further.
