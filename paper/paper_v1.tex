\subsection{Background}
\label{sec:background}

The extraction of textual information from images --- from scanned books to even receipts captured by smartphone cameras --- have become an important part of everyday life \cite{Investigation_binarization_techniques_unevenly_illuminated_document_images_alqudah_2015,review_degradation_image_enhancement_zhou_2023}. Any such extraction necessarily begins with the pre-processing of recorded images through techniques such as binarization, noise removal, and geometric transformation, a process generally referred to as \textit{document enhancement}. Numerous tools focused on this niche have been proposed. Although these approaches feature broad capabilities, they are often associated with substantial complexity on the user's end and entail considerable software dependencies. These dependencies restrict applicability, particularly in settings where binary size is constrained, such as mobile and web platforms. This limitation assumes added importance in a context where most digital media interactions now take place on smartphones and application development is increasingly oriented toward edge-computing models. Consequently, efficient and platform-independent implementations that exploit parallelization and other optimization techniques are of considerable importance for reducing processing time and enhancing user satisfaction.

Unlike the existing tools and libraries described, our approach is designed to be minimalistic, extensible, and user-friendly. Instead of being a comprehensive image processing library or OCR engine, we focus on a lightweight, modular image enhancement pipeline, focused on printed and handwritten text, written entirely in C++ with only \href{https://www.cimg.eu/}{CImg} as a dependency, which is itself is lightweight has no further dependencies \cite{cimg}. Image Text Enhancer (ITE) has a lightweight build footprint, requiring only three build dependencies: OpenMP, X11, and JPEG. We produce a Linux executable of 1.7 MiB.

We systematically optimize and improve our baseline implementation and evaluate the performance gains through various experiments. 
Compared to OpenCV and ImageMagick, our work is characterized by its deliberate focus on image enhancement and compact architecture. Unlike Tesseract and Leptonica, our pipeline is not integrated into an OCR workflow and limited by low-level primitive functions, but offers a configurable standalone solution that is ready for immediate use. Our pipeline is not exclusively specialized in the post-processing of scanned documents, but is designed as a general image enhancement pipeline that can be used for a variety of applications.
In addition, our licensing model (MIT-0 and CeCILL-C) ensures its use in research and commercial applications. By combining minimal dependencies, a modular architecture, portability through C++ and OpenMP, and a commercially usable licence, our pipeline fills a gap between large-scale, general-purpose image processing libraries and highly specialized tools for document enhancement. In a proof of concept, we show that our approach can be executed on WASM platforms with only minor performance deficits.

\section{Pipeline}
\label{sec:pipeline}
  
The pipeline is designed to be flexible and modular, enabling users to activate, deactivate and configure each step as required. Each step is designed to run independently of the others, except for the conversion to grayscale step \ref{pl:grayscale}, which forms the basis for subsequent operations. The steps in the pipeline are as follows:


\begin{algorithm}
  \caption{Image Text Enhancer Pipeline}
  \label{alg:enhance}
  \begin{enumerate}
  \item Grayscale Conversion
  \item Deskew
  \item Contrast Enhancement
  \item Denoise: (Adaptive) Gaussian and Median Blur
  \item Binarization: Otsu, Sauvola, Bataineh
  \item Morphological Operations
  \item Colour passthrough
\end{enumerate}
\end{algorithm}

Several algorithms are implemented in the steps \ref{pl:denoising}, \ref{pl:binarization} and \ref{pl:morphological}. As these have different effects on the result, users can select the algorithm best suited to their application. For instance, Bataineh's adaptive binarisation method with automatic window adjustment \cite{adaptive_thresholding_methods_bataineh_2011} is helpful when binarising historical, handwritten documents with uneven writing and lighting, whereas Otsu's method \cite{Otsu1979} is a good choice for binarising typewritten documents with uniform font and lighting.
Standard configurations are provided for each step, enabling the pipeline to be used straight away. These configurations deliver good results for most use cases and are based on recommendations from the authors. This means that the pipeline can be used immediately, without the need for additional information. However, users can also customise the configurations to optimise the pipeline and improve their specific results.
The pipeline methods \ref{pl:grayscale} - \ref{pl:color} can be used individually by users, as a library in their own program for their specific use case.

We provide a command-line application that can be used independently of a program to access the pipeline. This enables the pipeline to be configured and used directly via the command line.

\subsection{Parallelization and Concurrency}
\begin{comment}
  - Grayscale: parallel for, for pixel processing loop
  - Contrast Enhancement: parallel for unsing thread-local buffers to avoid false sharing/locking
  - Deskewing: logic split: split detection and application. Angle is detected once and then applied to the gray and color image
\end{comment}
To achieve better performance on multicore processors, the individual steps are parallelized using OpenMP. \textit{parallel for collapse(3)} environments are mainly used for the pixel processing loops. However, some binarization algorithms also use other parallelization strategies such as the reduction clauses \textit{reduction(min)} and \textit{recuction(max)}. When implementing parallelization, care is taken to ensure that no race conditions or false sharing occur. For example, thread-local buffers are used for contrast enhancement to avoid false sharing and locking. In the deskewing logic, detection and application of the correction are separated. The angle is detected once and then applied to the grayscale and color image to avoid unnecessary calculations. Intuitively, this was done twice.

\subsection{Memory Management and Allocation}
\begin{comment}
  - Grayscale: pointer swapping instead of copying the image for each step -> no memory tests were conducted
  - Adaptive Gaussian Blur: removed std::vector allocations inside hot loop and using "ping-pong" strategie where image acted as destination buffer, swapping pointers at the end
  - Adaptive Median Filter: moving of buffer allocations outside of the pixel loop
  - Suavola Binarization: removing of the entire 8-bit image to "CImg<double>" twice. No uses the lightweigt fused vector builder
  - CORE Utilites: returning pre-allocated buffers instead of heavy objects
\end{comment}
To prevent increased memory consumption and long runtimes, the image is not copied at each step. Instead, the steps work in-place on the input image. For this purpose, a pointer is passed at each step. This approach is continued in grayscale conversion by replacing the final deep copy task of the image with a pointer swap of the original image pointer with the new one. 
Memory optimizations were made in further steps. For example, in the adaptive Gaussian blur filter and the adaptive median filter, the buffer allocations are removed from the pixel processing loop to reduce the number of memory allocations. Sauvola's binarization algorithm uses a lightweight fused vector builder instead of the \textit{8-bit CImg<double>} image structure. Similarly, the core utilities return preallocated buffers instead of heavy objects.

\subsection{Algorithmic Complexity Reduction}
\begin{comment}
  - Batainhe Binarization: using integral images to calculate balck and red pixels instead of iterating over the complete window for each pixel -> reduced from windowsize * window size to 4 operations per pixel -> reduzierung von O(N \cdot W^2) zu O(N) 
  - Morphological Operations: decomposing the 2d squre kernel operation intwo two 1d kernel operations (horizontal and vertical) -> reducing the complexity from O(K^2) to O(2K)
  - Morphological Operations: implementation of "sliding Window Min/max" algorithm using a monotonic queue -> calculation for any windowd size in amortized O(1) constant time
  - Deskewing: replaced "rotate and project" method with a "radon transform". Instead of moving pixels, we extract foreground coordinates and mathematically project them
\end{comment}
Several different algorithms are implemented in the pipeline. Some of these are optimized through algorithmic improvements to reduce complexity. One example is Bataineh binarization, which reduces the number of operations per pixel from $O(W^2)$ to $O(1)$ by using integral images to calculate the number of black and red pixels in the window instead of iterating over the entire window. In morphological operations, the complexity is reduced from $O(K^2)$ to $O(2K)$ by decomposing the 2D kernel operations into two 1D kernel operations for the horizontal and vertical dimensions. In addition, the sliding window min/max algorithm is implemented with a monotonic queue to enable computation for arbitrary window sizes in amortized $O(1)$ time. In the deskewing logic, the “rotate and project” method is replaced by a “Radon transform” method. Instead of moving the pixels, the coordinates of the foreground are extracted and mathematically projected.

\subsection{Loop level Optimizations}
\begin{comment}
  - Contrast Enhancement: precalculate the linear strecht formula and save in Lookup-Table. main loop performs O(1) lookup insted of floating point calculations
  - Contrast Enhancement: replacing if/else branching in pixel loop with std::min clamping, alowing the compiler to auto-vectorize using SIMD -> no assembly checks were done
  - Sauvola Binarization: branchless lookup
\end{comment}
To reduce the runtime of pixel processing loops, logical optimizations are made to the main loops that iterate over the pixels. In contrast enhancement, for example, the linear stretching formula is calculated in advance and stored in a lookup table. As a result, the main loop performs an $O(1)$ query per loop iteration instead of floating-point calculations. In addition, the if/else branch in the pixel processing loop is replaced by std::min clamping. This allows the compiler to auto-vectorize the loop with SIMD. In Sauvola binarization, a branchless lookup is also implemented to improve performance.

\subsection{Architectial Refactoring}
\begin{comment}
  - Deskewing: logic split: split detection and application. Angle is detected once and then applied to the gray and color image
  - CORE Utilites: returning pre-allocated buffers instead of heavy objects
\end{comment}
In some cases, architectural changes were also made to improve performance. For example, in the deskewing logic, detection and application of the correction were separated. The angle is first detected and then applied to the grayscale and color images to avoid unnecessary calculations. In the core utilities, pre-built buffers are returned instead of creating heavy objects. This improves performance.

To evaluate the performance of the pipeline with the optimizations, various experiments are conducted and the results are compared with the times of the intuitive implementation. The experiments examine the runtime when processing images of different sizes with and without parallelization.

The experiments assess three aspects: (1) single-threaded runtime versus image size, (2) eight-threaded runtime versus image size, and (3) strong-scaling behaviour with increasing core count for a large fixed image size.

For our measurements, we use a machine with an AMD Ryzen 9 9950X3D 16-core processor (32 hyperthreads) running Ubuntu 24.04.1 LTS with 64 GB of RAM. Each core has a base frequency of 4.3 GHz, with boost frequency of around 5.5 GHz. Care has been taken to limit the number of background processes, so as not to affect performance.

Test images are individual images from different OCR datasets. One test image is document text from the FUNSD dataset \cite{FUNSD_Jaume_2019}, which has been scanned and contains a lot of noise. Two additional test images are handwritten documents from the HTR ICFHR 2016 dataset \cite{HTR_Dataset_ICFHR_2016_Toselli}. Both of these images have a resolution of XxY and exhibit bleed-through, whereby text from other pages is visible. For colour documents, a test image from the LRDE Document Binarisation Dataset \cite{Efficient_Multiscale_Sauvola_Binarization_Lazzara_Geraud_2013} is used. This is a scanned excerpt from a French magazine of very good quality containing many small sections of text. It has a resolution of XxY. A test image from the KITA imaging programme Forum is used to demonstrate a combination of historical text, UI elements and colour. All images are provided as JPG files.

To test performance with larger images, the KITA image, which has an initial resolution of XxY, is upscaled to 2k, 4k, 8k and 16k using the ImageUpscaler tool \cite{imageupscaler}. All images are processed using the pipeline and the runtime is measured for each step. Additionally, the images are processed using one and eight threads to evaluate the effects of parallelisation.
